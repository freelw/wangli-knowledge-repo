# 工程协作规范

## 目的

这篇文档用于统一记录当前后端多仓库协作时的基本开发规范，避免在分支、主线同步和 PR 流程上反复出错。

## 适用范围

这套规范适用于：

1. `browser-manager`
2. `k8s-chrome-daemon`
3. `lexmount-k8s-manifests`
4. SDK 与 quickstart 仓库
5. 其他参与同一需求联动开发的仓库

## 核心原则

### 1. 永远不要直接推送 `main`

`main` 只用于跟踪远端 `origin/main`，不能直接在本地 `main` 上做需求开发提交。

所有需求开发都应：

1. 先同步最新 `origin/main`
2. 从干净的本地 `main` 拉开发分支
3. 在开发分支上提交改动
4. 通过 PR 合并回主线

### 2. 本地 `main` 必须保持干净

本地 `main` 的职责是对齐远端主线，而不是承载临时开发。

如果发现本地 `main` 和 `origin/main` 分叉，应先清理或重建本地 `main`，再继续拉分支。

### 3. 拉分支前先同步 `origin/main`

每次开始新需求前，都应先执行：

1. `git fetch origin`
2. `git pull --ff-only origin main`

目标是确保本地 `main` 和 `origin/main` 一致，再从这个状态拉开发分支。

## 分支命名规范

### 标准格式

与同一需求相关的每一个仓库，都应统一使用：

```text
wangli_dev_<日期>
```

其中 `<日期>` 使用 8 位日期，例如：

```text
wangli_dev_20260413
```

### 重名处理

如果当天同名分支已经存在，则在后面追加序号：

```text
wangli_dev_20260413_1
wangli_dev_20260413_2
```

### 为什么要统一命名

同一需求涉及多个仓库时，统一分支名有三个好处：

1. 便于联调和对照
2. 便于在工作总结里串起多仓库改动
3. 便于后续追溯某次需求对应的完整改动集合

## 多仓库联动开发流程

如果一个需求同时涉及多个仓库，建议按下面顺序操作：

1. 在每个仓库同步最新 `origin/main`
2. 从各自本地 `main` 拉出同名开发分支
3. 在各仓库分别提交改动
4. 分别推送远端分支
5. 分别创建 PR
6. 在工作总结中把这些分支和 PR 统一记录

## 推荐最小 Git 流程

### 新需求开始前

```bash
git fetch origin
git checkout main
git pull --ff-only origin main
git checkout -b wangli_dev_YYYYMMDD
```

### 开发完成后

```bash
git push -u origin wangli_dev_YYYYMMDD
```

然后基于远端分支创建 PR，而不是直接推主线。

## 常见错误

### 错误 1：直接在本地 `main` 上提交

问题：

1. 后续很难保证本地 `main` 干净
2. 容易和远端主线产生分叉
3. 会破坏“主线只跟踪远端”的约束

### 错误 2：拉分支前没同步主线

问题：

1. 新分支可能基于旧代码
2. 容易在联调时出现已修复问题重复出现
3. 多仓库之间会出现“同名分支但基线不同”的隐性风险

### 错误 3：同一需求在不同仓库使用不同分支名

问题：

1. 工作总结难以统一
2. PR 对照成本高
3. 回溯需求时容易漏仓库

### 错误 4：把本地 `main` 搞脏后继续滚动开发

问题：

1. 新需求起点不清晰
2. 容易把上一个需求的残留改动带进来
3. 排查问题时很难判断哪些改动属于当前需求

## 推荐实践

### 1. 把每次需求当成一次干净分支开发

不要在旧开发分支上不断堆新需求。

### 2. 多仓库需求统一记录分支名

在任务总结里明确记录：

1. 涉及哪些仓库
2. 每个仓库对应哪个分支
3. 哪些 PR 对应这次需求

### 3. 发布前再次确认分支来源

尤其在多仓库联动场景下，要确认所有开发分支都来自最新主线，而不是来源于旧需求分支。

## 当前结论

当前最重要的工程协作纪律只有四条：

1. 不要直接改或推 `main`
2. 本地 `main` 始终保持干净
3. 拉分支前一定同步最新 `origin/main`
4. 同一需求跨仓库统一使用同名开发分支
