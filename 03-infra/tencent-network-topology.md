# 腾讯云网络拓扑

## 目的

这篇文档用于沉淀当前和腾讯云网络相关的最小认知，重点回答：

1. `qcloud` 和 `qcloud-hk` 的网络骨架大致是什么
2. 排查跨区域访问、Grafana 入口、数据库访问时应该先看哪些对象
3. 当前已知的网络层风险点在哪里

## 当前结论先看

基于已有调研材料，当前可以先记住这几个结论：

1. `qcloud-hk` 的三层网络骨架已经基本对齐 `qcloud`
2. 香港环境的明显风险点不在 VPC Peering 或主路由缺失
3. 更需要警惕的是安全组、出口能力和区域镜像同步

## 主体网络对象

在腾讯云相关环境里，当前最常反复出现的是两类 VPC：

1. `browser-pool`
2. `Default-VPC`

常见理解方式：

1. `browser-pool` 更偏浏览器运行池
2. `Default-VPC` 更偏网关、数据库、基础服务
3. 两者之间通过 VPC Peering 和路由互通

## 当前已知的网络骨架

### `qcloud`

已知结构重点：

1. 有 `browser-pool`
2. 有 `Default-VPC`
3. 两者之间已有 Peering
4. 双向路由已经存在
5. 有 NAT 承担出口

### `qcloud-hk`

已知结构重点：

1. 同样有 `browser-pool`
2. 同样有 `Default-VPC`
3. 同样有 Peering
4. 主路由结构大体对齐南京
5. 但出口 EIP 数量和安全组策略与主区域不同

## 当前最值得优先看的风险点

### 1. 安全组

已有调研结论显示，香港环境更值得优先排查安全组，而不是先怀疑路由。

典型风险：

1. 跨 VPC 数据库访问未放通
2. 模板安全组放得过宽
3. 业务流量与管理流量规则混在一起

### 2. NAT / 出口

区域网络问题里，出口能力也是高频点：

1. 某区域 NAT EIP 数量不足
2. 出口地址和白名单预期不一致
3. 入口机 / SNAT / Kong 白名单链路不一致

### 3. 入口机到 Kong / Grafana 链路

运维类访问链路通常不是“直接访问 Grafana”，而是：

1. VPN / 入口机
2. Kong HTTPS NodePort
3. `ops-grafana` 路由
4. Grafana Service

这里常见问题是：

1. 入口机四层转发没配对
2. Kong `ip-restriction` 配置不一致
3. 环境白名单策略不同

## 遇到哪类问题先看哪层

### 场景 1：`qcloud-hk` 访问 PG 或内部服务异常

先看：

1. 安全组
2. Peering 两侧路由
3. 跨 VPC 允许源网段

### 场景 2：Grafana 入口异常

先看：

1. 入口机转发规则
2. Kong NodePort
3. `kong-init/config-*.js`
4. `ops-grafana` 的 `ip-restriction`

### 场景 3：区域环境看起来“代码一致但行为不同”

先不要先怀疑网络完全不通，优先确认：

1. 镜像是否同步到对应区域
2. 目标环境是否真的发布到新版本
3. 安全组 / 白名单是不是按区域分别配置

## 当前结论

腾讯云网络排查里，推荐默认优先级是：

1. 先看安全组和白名单
2. 再看 NAT / 入口 / 出口路径
3. 最后再看 Peering 和主路由是否缺失

因为从当前材料看，真正高频的风险点更多是“边界没配对”，而不是“骨架完全没搭起来”。
