# Browser 专项维度每周跟进流程

## 背景

本流程用于每周梳理 Browser 项目专项进展，将分散在 Notion 文档、晨报、周报、会议纪要、PR / issue 汇总、发布记录中的信息，汇总回固定的专项里程碑结构中。

目标不是重新写一份长报告，而是让读者快速判断：

- 当前有哪些 Browser 内核专项和云平台专项。
- 每个专项当前推进到哪个里程碑。
- 本周哪些里程碑或子点有实质进展。
- 哪些内容已完成、进行中、未开始或存在风险。
- 本周相对上一周发生了哪些变化。

Notion 规则源文档：

```text
每周跟进规则-prompt
https://www.notion.so/36ee7e998fb2809b95a7efe476e902a3
```

## 输入文档

每周先从以下三个入口读取结构和资料线索：

```text
Browser 内核专项信息
https://www.notion.so/36ce7e998fb2801d8131cfec03e1f7e6
```

```text
browser 云平台专项信息
https://www.notion.so/36ce7e998fb28003b1c1fde6ac2d399e
```

```text
Browser 文档导读（先看我）
https://www.notion.so/eef9ae77e87d4d479dd1c488471a30e9
```

## 输出文档

每周生成两个当前周 Notion 进展文档：

```text
Browser 内核专项本周进展 - YYYY-MM-DD
browser 云平台专项本周进展 - YYYY-MM-DD
```

这两个输出必须创建为 Notion 文档，并放在 Browser 专项维度相关空间下，便于后续按周归档、链接来源和人工阅读。

上一周的 `本周进展` 文档视为历史归档，不再覆盖。确有错误时只做事实修正，不把新周进展追加到旧文档中。

## 标准流程

### 1. 读取专项结构

先读取 `Browser 内核专项信息` 和 `browser 云平台专项信息`，提取以下结构：

- 当前有哪些专项。
- 每个专项负责人是谁。
- 每个专项当前处于哪个里程碑。
- 每个专项下有哪些完整里程碑。
- 正在进行的里程碑下有哪些跟踪子点。

注意：负责人、里程碑标题、子点标题以专项信息页为准，不要在周进展生成时随意改名或改负责人。

### 2. 确定本周搜索范围

通过 `Browser 文档导读（先看我）` 进入本周资料。搜索范围包括：

- Browser 周报中心。
- Browser 晨报中心。
- 相关会议纪要。
- 相关需求文档和技术方案。
- PR / issue 汇总和发布记录。
- 专项子文档及其下一级跟踪文档。

搜索方式以 `专项名称 + 当前里程碑 + 跟踪子点` 为主，不要只按日期粗略扫描。

### 3. 匹配本周进展

对每个专项逐项判断：

- 本周是否有该专项进展。
- 进展落在哪个里程碑。
- 进展对应哪个跟踪子点。
- 是否出现新的风险、延期或阻塞。
- 是否导致里程碑或子点状态变化。

所有生成内容必须限定在该专项当前里程碑范围内。只有当前里程碑及其跟踪子点相关的进展，才能写入本周进展正文。

如果搜索到的资料属于其他里程碑：

- 已结束里程碑：只在需要说明状态回归或风险复燃时记录，否则不写入本周进展。
- 未开始里程碑：只作为候选线索或后续待办，不写成当前进展。
- 无法判断归属：标记为 `待确认归属`，不要直接并入当前里程碑。

每条进展必须附来源文档。无法确认来源时，写 `来源待补充`，不要写成确定结论。

### 4. 生成本周进展文档

分别生成内核专项和云平台专项本周进展。每个文档按专项展开。

每个专项必须列出全部里程碑，而不是只列本周有进展的里程碑。这样每周结构保持稳定，方便对比上周和本周。

### 5. 标记进度状态

统一使用以下状态标记：

- `✅ 已结束`
- `🟡 进行中`
- `⚪ 未开始`
- `🔴 风险 / 延误`

里程碑层级必须标状态。正在进行的里程碑下，跟踪子点也必须标状态。

已结束或未开始的里程碑可以只写一行，避免文档膨胀。正在进行的里程碑需要展开子点。

### 6. 编写正在进行的里程碑

正在进行的里程碑下，每个子点按以下格式写：

```text
- `🟡 子点名称`：本周进展一句话。来源：文档链接
```

写法要求：

- 摘要尽量短。
- 只写本周事实进展。
- 不复制长段原文。
- 不把长期背景写成本周进展。
- 没有新增进展的子点可以写 `本周无新增`。

### 7. 归档历史周

生成新一周文档前，确认上一周文档已经成为历史记录。

建议保留历史文档名称中的日期，不重命名为通用标题。新文档只承载当前周内容，避免一个页面持续膨胀。

## 输出模板

输出结构只保留专项、里程碑和当前里程碑展开子点。

不要生成以下区块：

- `Prompt 来源`
- `数据来源`
- `TL;DR`
- 末尾独立的 `本周风险`

```text
# Browser 内核专项本周进展 - YYYY-MM-DD

## 专项 1：专项名称

**负责人**：姓名
**当前阶段**：Mx 里程碑名称

### 里程碑进度

- `✅ M1 xxx`：已结束。
- `🟡 M2 xxx`：进行中。
  - `🟡 子点 A`：本周进展一句话。来源：文档链接
  - `⚪ 子点 B`：本周无新增。来源：文档链接
- `⚪ M3 xxx`：未开始。
```

云平台专项文档使用相同结构。

## 质量红线

以下情况需要返工：

- 只写本周有进展的专项，遗漏无进展专项。
- 没有列出全部里程碑。
- 正在进行的里程碑没有展开子点。
- 有进展的子点没有来源文档。
- 把非当前里程碑范围内的内容写成本周进展。
- 把长期背景写成本周进展。
- 生成 `Prompt 来源`、`数据来源`、`TL;DR` 或末尾独立 `本周风险` 等额外区块。
- 状态标记不一致。
- 文档结构每周变化太大，无法与上周快速对比。
- 周进展文档持续追加历史内容，导致页面不可读。

## 执行建议

- 先固定专项结构，再补本周内容。
- 先写事实状态，再写风险和下周动作。
- 先覆盖所有专项，再打磨重点专项。
- 对责任人、里程碑标题、子点标题保持克制，只在源专项信息页明确调整后同步。
- Notion 页面排版优先使用短标题、短列表和链接，避免复杂表格和大段引用。
